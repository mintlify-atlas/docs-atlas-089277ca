---

The CrowdSec Central API (CAPI) is the global network backbone that makes CrowdSec a crowd-powered security engine. When your LAPI is connected to CAPI, it participates in two-way threat intelligence exchange: your local detection signals are contributed to the network, and in return your installation receives a continuously updated community blocklist of confirmed malicious IPs. Neither raw logs nor personal data are shared — only aggregated, anonymized signals.

## How CAPI works

```
[CrowdSec agent] --> [LAPI] --push signals--> [CAPI]
                       |   <--pull blocklist--    |
                  [bouncers]                  [other nodes]
```

LAPI manages the CAPI connection on behalf of all local agents and bouncers. Agents never contact CAPI directly.

<CardGroup cols={2}>
  <Card title="Push (every ~10 seconds)" icon="arrow-up">
    LAPI batches alert signals from local agents and sends them to CAPI. Only certified, non-simulated scenarios are shared by default.
  </Card>
  <Card title="Pull (every ~2 hours)" icon="arrow-down">
    LAPI fetches the community blocklist and any subscribed blocklists from CAPI and stores them as local decisions. Bouncers receive these decisions via the stream endpoint.
  </Card>
</CardGroup>

## Registering with CAPI

<Steps>
  <Step title="Register your instance">
    Run the following command on the host where LAPI is installed. This creates a machine account on the CrowdSec network and writes credentials to `online_api_credentials.yaml`.

    ```bash
    cscli capi register
    ```

    You can optionally provide a name to identify your instance in the console:

    ```bash
    cscli capi register --name "my-production-server"
    ```
  </Step>

  <Step title="Verify the connection">
    Check that LAPI can reach CAPI and that credentials are valid:

    ```bash
    cscli capi status
    ```

    A successful response looks like:

    ```
    INFO[2026-04-23T10:00:00Z] Loaded credentials from /etc/crowdsec/online_api_credentials.yaml
    INFO[2026-04-23T10:00:00Z] Trying to authenticate with Central API (CAPI)
    INFO[2026-04-23T10:00:00Z] You can successfully interact with Central API (CAPI)
    ```
  </Step>

  <Step title="Restart CrowdSec">
    LAPI reads the CAPI credentials at startup. After registering, restart the service to activate the connection:

    ```bash
    sudo systemctl restart crowdsec
    ```
  </Step>
</Steps>

## Credentials file

LAPI stores CAPI credentials in the file pointed to by `api.server.online_client.credentials_path` in `config.yaml`. The default path is `/etc/crowdsec/online_api_credentials.yaml`.

```yaml
url: https://api.crowdsec.net/
login: a1b2c3d4e5f6a1b2c3d4e5f6  # machine ID on the CrowdSec network
password: a-strong-randomly-generated-password
papi_url: https://papi.api.crowdsec.net/  # PAPI URL for console-managed decisions
```

<Warning>
  Do not edit this file manually. Use `cscli capi register` to generate it and `cscli capi status` to verify it.
</Warning>

## Controlling what is shared

You can fine-tune signal sharing and blocklist pulling in `config.yaml`:

```yaml
api:
  server:
    online_client:
      sharing: true       # set to false to disable all signal sharing
      pull:
        community: true   # pull the community blocklist
        blocklists: true  # pull subscribed third-party blocklists
```

Individual signal types can also be controlled from the console (see below) or by setting `simulated: true` in a scenario to prevent its alerts from being shared.

## What data is shared

CrowdSec is designed to share as little as necessary:

| Shared | Not shared |
|---|---|
| Source IP or CIDR range | Raw log lines |
| Scenario name and version | Usernames, passwords, URLs |
| Aggregated event count | Application payload data |
| Alert start/stop timestamp | Private IP addresses |
| Country, ASN (if available) | Any personal data |

<Note>
  Private IP ranges (RFC 1918) are never forwarded to CAPI. Simulated alerts are also excluded.
</Note>

By default, signals from community-certified scenarios are shared. You can additionally opt in to sharing signals from:

- **Custom scenarios** — scenarios you wrote yourself
- **Tainted scenarios** — official scenarios you have locally modified
- **Manual decisions** — IPs you banned with `cscli decisions add`

Configure these options in the CrowdSec console under **Settings → Security Engine → Sharing**, or with `cscli console enable/disable`.

## Console integration

Enrolling your instance in the [CrowdSec console](https://app.crowdsec.net) provides a management dashboard, access to premium blocklists, and the ability to push decision changes to your LAPI remotely.

```bash
# Enroll using the key shown in the console under "Security Engines"
cscli console enroll <enrollment-key>

# Check current enrollment and console options
cscli console status
```

After enrollment, decisions pushed from the console travel through the PAPI (Premium API) channel, which uses the `papi_url` credential to establish a persistent connection.

## CAPI intervals

| Operation | Default interval |
|---|---|
| Push signals | Every 10 seconds |
| Pull community blocklist | Every 2 hours |
| Send metrics | Every 30 minutes |
| Send usage metrics | Every 30 minutes |

Intervals include a small random delta to spread load across the network.

## Disabling CAPI

To run CrowdSec in a fully air-gapped mode, remove or leave empty the `online_client` section in `config.yaml`. LAPI will log `push and pull to Central API disabled` at startup and operate entirely on local signals and decisions.
