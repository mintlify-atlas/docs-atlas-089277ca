---
title: "CrowdSec quickstart: install, configure, and block"
description: "Install CrowdSec on Linux, verify the service, add a bouncer to enforce decisions, and view your first alert — all in under ten minutes."
---

CrowdSec is installed as a system service that reads your log sources, detects attacks using community-maintained scenarios, and shares signals with the global network. This guide walks you through installing the security engine, confirming it is running, adding a bouncer to enforce bans, and inspecting your first alert.

<Steps>

<Step title="Add the CrowdSec repository and install">

CrowdSec publishes packages for both Debian/Ubuntu-based and RHEL/CentOS-based distributions through an official repository.

<CodeGroup>

```bash apt (Debian / Ubuntu)
curl -s https://packagecloud.io/install/repositories/crowdsec/crowdsec/script.deb.sh | sudo bash
sudo apt-get install crowdsec
```

```bash yum / dnf (RHEL / CentOS / Fedora)
curl -s https://packagecloud.io/install/repositories/crowdsec/crowdsec/script.rpm.sh | sudo bash
sudo yum install crowdsec
```

</CodeGroup>

<Note>
The install script configures the packagecloud repository and installs the `crowdsec` package along with `cscli`, the command-line management tool. No separate install step is needed for `cscli`.
</Note>

After installation, the service starts automatically and begins reading the log sources detected on your system (for example, `/var/log/auth.log` on Debian-based systems).

</Step>

<Step title="Verify the service is running">

Check that the CrowdSec service is active and that the Local API is listening on `127.0.0.1:8080`:

```bash
sudo systemctl status crowdsec
```

You should see `active (running)` in the output. You can also use `cscli` to confirm the engine has registered itself:

```bash
sudo cscli machines list
```

A healthy installation shows one machine with a recent heartbeat timestamp. To view what log sources are being read and which parsers and scenarios are loaded:

```bash
sudo cscli metrics
```

<Tip>
If the service fails to start, check `/var/log/crowdsec.log` for details. The most common cause is a conflict with an existing process on port 8080.
</Tip>

</Step>

<Step title="Install a bouncer">

CrowdSec detects threats and records decisions, but enforcement is done by **bouncers** (also called Remediation Components). A bouncer reads decisions from the Local API and applies them — for example, by dropping packets at the firewall level.

Install the firewall bouncer, which integrates with `iptables` or `nftables`:

<CodeGroup>

```bash apt (Debian / Ubuntu)
sudo apt-get install crowdsec-firewall-bouncer-iptables
```

```bash yum / dnf (RHEL / CentOS / Fedora)
sudo yum install crowdsec-firewall-bouncer-iptables
```

</CodeGroup>

The bouncer registers itself with the Local API automatically. Verify it appears in the bouncer list:

```bash
sudo cscli bouncers list
```

You should see `crowdsec-firewall-bouncer` listed with a recent last pull time. From this point, any IP that triggers a ban decision is automatically blocked at the firewall level.

<Note>
Other bouncers are available for Nginx, Traefik, Caddy, AWS WAF, and more. Browse the full list at [hub.crowdsec.net](https://hub.crowdsec.net).
</Note>

</Step>

<Step title="View your first alert">

As CrowdSec processes your logs, it raises alerts when a scenario threshold is exceeded — for example, multiple failed SSH authentication attempts from the same IP.

List recent alerts:

```bash
sudo cscli alerts list
```

Each alert shows the scenario that fired, the offending IP, the scope, and the timestamp. To see the full detail for a specific alert, use its ID:

```bash
sudo cscli alerts inspect <alert-id>
```

To see the active ban decisions currently enforced by your bouncers:

```bash
sudo cscli decisions list
```

<Tip>
You can also view alerts and decisions with a filter. For example, to show only SSH brute-force alerts: `sudo cscli alerts list --scenario crowdsecurity/ssh-bf`
</Tip>

</Step>

</Steps>

## What happens next

With CrowdSec running, your server is already benefiting from the community blocklist — a curated list of known-malicious IPs that is pushed to your instance from the Central API. Every alert your instance generates is also (anonymously) shared back, protecting other users in the network.
